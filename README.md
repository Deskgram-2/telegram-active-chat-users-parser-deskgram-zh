## 模块简介

| 参数 | 内容 |
|---|---|
| 核心任务 | 收集在 Telegram 聊天中真实发言的用户 |
| 关键区域 | 聊天列表、过滤、统计、日志、结果 |
| 适用场景 | 讨论型 niche、暖受众准备、触达漏斗 |
| 自然下一步 | 私信群发、邀请增长、后续细分 |
| 配套 parser 路线 | 评论受众收集 |

## 模块能力

- 从 Telegram 活跃聊天中整理发言用户；
- 围绕选定聊天来源工作，而不是宽泛随机收集；
- 过滤并结构化结果；
- 展示执行进度和日志；
- 为触达和增长流程准备更实用的受众层。

## 快速开始

1. 选择讨论活跃、主题清晰的聊天来源。
2. 把这些聊天加入模块。
3. 配置限制与过滤规则。
4. 运行任务并检查结果。
5. 将整理好的受众继续送入沟通、invite 或细分流程。

## 通常下一步怎么走

- [私信群发](https://github.com/Deskgram-2/telegram-direct-messaging-deskgram-zh)，如果下一步是建立一对一联系。
- [邀请模块](https://github.com/Deskgram-2/telegram-invite-tool-deskgram-zh)，如果活跃用户会进入增长流程。
- [受众收集](https://github.com/Deskgram-2/telegram-audience-parser-deskgram-zh)，如果你还需要更宽的收集路线。
- [评论受众收集](https://github.com/Deskgram-2/telegram-comment-audience-parser-deskgram-zh)，如果你还想补上帖文互动信号。
- [任务管理器](https://github.com/Deskgram-2/telegram-task-manager-deskgram-zh)，如果你要把整条链路放在一个控制层里。

## 场景是怎么工作的

### 聊天来源层

先给出那些已经能代表主题、而且讨论真实存在的聊天。来源质量会直接决定最终受众层的相关性。

### 过滤与清洗

过滤规则可以减少噪音，让结果更适合后续触达或增长动作。

### 路由进漏斗

收集后的用户可以继续送入私信、invite 或更细的受众分层流程。

## 特别适合什么时候用

- 当 niche 本身就高度依赖聊天讨论时；
- 当你更看重真实发言行为，而不是宽泛成员数量时；
- 当回复质量比最大覆盖面更重要时；
- 当漏斗入口来自活跃社区行为时。

## 为什么它比宽泛收集更强

| 宽泛收集 | Deskgram 2 聊天活跃用户收集 |
|---|---|
| 互动信号较弱 | 发言本身就是明确活跃信号 |
| 很难理解社区语境 | 聊天参与更能体现真实行为 |
| 结果可能过宽 | 以聊天来源为中心更聚焦 |
| 后续触达不够精准 | 更适合讨论驱动型漏斗 |

## 该选哪个：聊天活跃用户收集还是评论受众收集

| 如果你的目标是 | 更适合哪个 |
|---|---|
| 收集聊天里真实发言的用户 | [聊天活跃用户收集](https://github.com/Deskgram-2/telegram-active-chat-users-parser-deskgram-zh) |
| 收集帖文评论区下活跃的用户 | [评论受众收集](https://github.com/Deskgram-2/telegram-comment-audience-parser-deskgram-zh) |
| 组一层多信号的暖受众 | 两条路线一起使用 |
| 围绕讨论型社群做受众准备 | 聊天活跃用户收集 |

## 场景 FAQ

### 它什么时候比普通 parser 或评论 parser 更强？

当最重要的信号来自聊天里的真实发言，而不是成员关系或评论区互动时。

### 这批用户通常送去哪里？

通常会进入 [私信群发](https://github.com/Deskgram-2/telegram-direct-messaging-deskgram-zh)、[邀请模块](https://github.com/Deskgram-2/telegram-invite-tool-deskgram-zh) 或后续分层流程。

### 什么最影响最终结果？

聊天来源本身的质量、真实活跃度，以及你是否已经明确收集后的下一步动作。
